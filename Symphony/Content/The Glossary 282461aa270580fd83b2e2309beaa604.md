## A

**ACA (Access Control Architecture)**

A security framework for selective internal API access. Allows trusted sources (Microsoft, enterprise partners, vetted developers) to access internal APIs while maintaining security boundaries from untrusted community extensions. Enables dual library systems where enterprise and community extensions operate with different permission levels.

**ADD (Agent-Driven Development)**

A development paradigm where autonomous AI agents drive the software creation process while humans provide vision, guidance, and creative direction. Replaces the traditional model of developers writing every line of code manually. In ADD, you become a composer orchestrating intelligent systems rather than a coder writing every function.

**Addon (Motif)** üß©

User-facing extensions that enhance Symphony's visual and interactive experience. Part of The Grand Stage (UFE) and External Assembling. Includes custom dashboards, specialized editors, developer tools, UI enhancements, and themes. Addons shape how developers visualize, debug, and interact with their projects. **Low policy focus** - primarily visual enhancements with minimal security concerns.

**AIDE (AI-First Development Environment)**

A new class of development system where AI agents are the primary actors, not supplementary tools. Unlike traditional IDEs that embed AI as features, an AIDE places intelligent orchestration at its core. Symphony is the first true AIDE, where AI doesn't just assist‚Äîit orchestrates the entire development lifecycle.

**Arbitration Engine** ‚öñÔ∏è

One of The Pit's five Rust-powered infrastructure extensions (Internal Assembling). Makes intelligent, fair decisions about resource allocation when multiple workflows compete for limited resources. Considers business value, user impact, cost efficiency, and learning value to resolve conflicts. Ensures equitable distribution while respecting priorities and constraints.

**Artifact**

Structured outputs produced by models and extensions throughout Symphony's workflow. Examples include JSON plans, configuration files, documentation, CSV backlogs, and code files. Artifacts form the communication layer between different components in the orchestration pipeline.

**Artifact Store** üíæ

One of The Pit's five infrastructure extensions (Internal Assembling). Symphony's institutional memory that stores, versions, and manages every piece of data flowing through workflows. Tracks quality, relationships, and lineage of all artifacts. Provides intelligent retrieval with predictive preloading and maintains complete audit trails for compliance.

**Assembling**

The complete extension system architecture in Symphony, divided into Internal Assembling (The Pit - infrastructure extensions) and External Assembling (The Grand Stage - user-facing extensions). Represents the modular approach to building Symphony's capabilities through extensions rather than built-in features.

**BiE (Built-in Extensions)**

Extensions that ship with Symphony's core installation. Includes essential functionality that every user needs immediately, like basic file operations, terminal integration, and fundamental UI components. Distinguished from community extensions which are installed separately.

**Conductor** üé©

Symphony's intelligent maestro‚Äîa Python-based Reinforcement Learning model trained through Function Quest Game. The microkernel core that orchestrates all AI agents, manages workflows, and makes strategic decisions about when to activate models and how to handle problems. **Conductor Core**

**Coordinator Model** üîπ

## D

**DAG (Directed Acyclic Graph)**

The workflow structure that DAG Tracker uses to manage dependencies between tasks. Ensures operations execute in the correct order while identifying opportunities for parallel execution.

**DAG Tracker** üìà

One of The Pit's five infrastructure extensions (Internal Assembling). Symphony's workflow cartographer that maps dependencies, determines optimal execution order, manages state, and handles recovery. Creates checkpoints and rollback strategies, identifies bottlenecks, and learns optimal execution patterns from historical data.

**DEA (Dual Ensemble Architecture)**

Symphony's architectural pattern of two distinct extension layers: The Pit (Infrastructure as Extensions / Internal Assembling) for performance-critical operations, and The Grand Stage (User-Faced Extensions / External Assembling) for creative functionality. Each layer uses different execution models optimized for their specific needs.

## E

**Editor Model** üë®‚Äçüíª

One of Symphony's seven orchestration models. Generates actual code files based on instructions from upstream models. Handles syntax, formatting, best practices, and produces production-ready source code. The final translation step from design to implementation.

**Enhancer-Prompt Model** üé§

One of Symphony's seven orchestration models. The first model in the pipeline that enhances raw user prompts into detailed technical blueprints. Clarifies intent, adds technical context, and structures requirements for downstream models. Transforms vague ideas into actionable specifications.

**EPP (Execution Policy Playbook)**

A  framework managing execution policies for all community-built extensions. Applies shared policies across extension types while providing specialized enforcement based on category-specific requirements. Ensures reliability, security, and performance across the entire extension ecosystem. Defines how extensions are monitored, limited, and controlled during execution.

**Exokernel Architecture**

One of the five kernel architecture patterns available for system design. Provides minimal kernel duties with raw resource allocation. Extensions communicate directly with each other and build their own abstractions. Offers ultimate performance and customization but requires more complex development. Perfect for research environments and experimental systems where maximum flexibility outweighs stability concerns.

**Extension**

A modular component that adds functionality to Symphony. Can be infrastructure (IaE in The Pit / Internal Assembling) or user-facing (UFE on The Grand Stage / External Assembling). Extensions communicate through the Orchestra Kit using either in-process calls (The Pit) or IPC (user extensions).

**Extension Categories**

Three types of user-facing extensions with different policy requirements:

- **üéª Instruments**: AI/ML models (High policy focus - external dependencies, API calls, model hosting)
- **‚öôÔ∏è Operators**: Utility functions (Medium policy focus - lightweight processing, data transformation)
- **üß© Motifs (Addons)**: UI components (Low policy focus - visual enhancements, themes, dashboards)

**External Assembling**

The user-facing extension layer, also known as The Grand Stage (UFE). Where community developers build Instruments (AI models), Operators (utilities), and Addons/Motifs (UI enhancements). Extensions run out-of-process for safety and isolation. The creative, innovative layer where the community builds specialized functionality.

**External Ensemble**

Another term for External Assembling or The Grand Stage - the collection of community-driven extensions that run outside the core process.

## F

**Feature Model** üéº

One of Symphony's seven orchestration models. Breaks down high-level functionality into epics and tasks. Creates structured backlogs that guide the development process. Transforms abstract requirements into concrete work items.

**FFI (Foreign Function Interface)**

The bridge layer that allows Python Conductor Core to call Rust extensions in The Pit with near-zero overhead. Enables the performance of native Rust with the intelligence of Python RL models.

**FQ (Function Quest)**

Short reference to Function Quest Game‚Äîthe training system where the Conductor learns orchestration logic.

**FQG (Function Quest Game)**

A logic-based puzzle game where players solve levels by deducing the correct sequence of function calls. The training ground where Symphony's Conductor model learns orchestration strategies through reinforcement learning. Each level teaches reasoning, dependency management, and optimal decision-making.

**FQM (Function Quest Maestro)**

The Conductor's identity within Function Quest‚Äîthe orchestration expert trained through gameplay. Represents the Conductor's mastery of treating models as callable functions in complex logical workflows.

**FQT (Function Quest Training)**

The complete training methodology using Function Quest Game to develop the Conductor's orchestration intelligence. Involves progressive difficulty levels, performance tracking, reinforcement learning feedback, and continuous improvement through gameplay scenarios.

The public marketplace where innovation performs.

## H

A visual node-based interface for designing and debugging AI agent interactions. Part of Symphony's orchestration tools that lets you see workflow execution in real-time, identify bottlenecks, and optimize your melodies.

**Hybrid Kernel Architecture**

One of the five kernel architecture patterns. Balances performance and flexibility by placing critical components in kernel space and security-sensitive components in user space. Offers the best of both monolithic and microkernel approaches with strategic component placement.

**H¬≤A¬≤ (Harmonic Hexagonal Actor Architecture)**

Symphony's foundational architectural model. Combines Hexagonal Architecture (ports/adapters) with Actor-based concurrency (autonomous Instruments), all orchestrated by the Conductor Model. Ensures strict boundary purity, adaptive flow control, and musical coherence across all orchestration layers.

## I

**IaE (Infrastructure as Extensions)**

The architectural philosophy where even core infrastructure runs as extensions. Embodied by The Pit's five Rust-powered components (Internal Assembling) that provide Symphony's foundation. These extensions use in-process execution for maximum performance.

**IDE (Integrated Development Environment)**

Traditional development tools (like VSCode, IntelliJ) built around human developers as central operators. Symphony transcends this with AIDE‚Äîwhere AI is central, not peripheral.

**IEC (Inter-Extension Communication)**

The safe, validated communication form that enables extensions different layers of communication within Symphony

**In-Process Execution**

The execution model used by The Pit (Internal Assembling) where extensions run within the same process as the Conductor. Enables microsecond-level performance through direct function calls and shared memory. Used for performance-critical infrastructure that requires nanosecond response times.

**Instrument** üéª

AI/ML models integrated as extensions on The Grand Stage (External Assembling). Can be hosted models (GPT, Claude), local models, or custom fine-tuned models. Instruments bring intelligence to workflows and can be orchestrated by the Conductor in both Maestro Mode (automatic) and Manual Mode (user-controlled). **High policy focus** due to external dependencies, API calls, and model hosting requirements.

**Internal Assembling**

The infrastructure extension layer, also known as The Pit (IaE). Contains five privileged Rust-powered extensions: Pool Manager, DAG Tracker, Artifact Store, Arbitration Engine, and Stale Manager. These extensions run in-process with the Conductor for maximum performance and form Symphony's unshakable foundation.

**Internal Ensemble**

Another term for Internal Assembling or The Pit - the collection of core infrastructure extensions that run in-process.

**IPC (Inter-Process Communication)**

The communication mechanism used by user-facing extensions (UFE / External Assembling). Provides process isolation for safety while maintaining acceptable performance (0.1-0.5ms latency). The IPC Bus routes messages between the Conductor and isolated extension processes.

**IPC Bus**

A high-performance Rust crate that manages all communication between the Conductor and out-of-process extensions. Implements multiple transport strategies (Unix sockets, shared memory, named pipes) optimized for different message patterns.

## K

**Kernel Architecture**

The fundamental organizational pattern for Symphony's core. Symphony uses a Microkernel Architecture‚Äîminimal privileged core (Conductor) with everything else as extensions. Provides maximum modularity, reliability, and extensibility.

## M

**Maestro Mode** üé∂

One of Symphony's three operational modes. Ideal for full application creation from a single prompt. Uses complete orchestration with all seven models working together in a coordinated pipeline. The Conductor automatically selects and blends AI models for optimal results, making all configuration and execution decisions based on learned patterns and user context. Best for starting new projects from scratch.

**Manifest**

Every extension's contract with Symphony. A TOML file that declares what the extension does, what permissions it needs, what capabilities it provides, and how it should be executed. The Orchestra Kit uses manifests to generate configuration UIs, manage dependencies, and enforce security.

**Manual Mode**

An orchestration mode where users have complete control over model selection, configuration, and execution. For developers who want explicit control over which models run and how.

**MCCF (Model Criticality Classification Framework)**

A system that automatically classifies models as Essential, Recommended, or Optional through balanced objective analysis. Combines practical pattern matching with adaptive weighted scoring to determine model importance in workflows. Uses transparent reasoning that balances automation with human understanding. Helps Symphony make intelligent decisions about which models are truly necessary for a given task.

**Melody**

A user-created or Conductor-generated workflow in Symphony‚Äîa visual, composable sequence of extensions from the Orchestra Kit, orchestrated to automate development tasks. Melodies can be manually composed by the user or automatically generated by the Conductor Model based on user requirements (typically introduced as a prompt from the user to the Conductor in chat). Each Melody is composed on the Harmony Board as a ‚Äúmusical score,‚Äù where nodes (Instruments, Operators, Addons) are connected, initiated by triggers (manual or scheduled), and configured for behavior (parameters, error handling). The Conductor manages execution, routes events, dispatches tasks, and handles errors (retry, fallback, skip). Extensions produce artifacts (inputs, intermediates, outputs, metadata) for auditing and flow chaining.

**Microkernel Architecture**

Symphony's chosen kernel architecture pattern. Features an ultra-minimal core (Conductor) with maximum modularity‚Äîall other components run as extensions in user space. Provides exceptional reliability through isolated failures, easy maintenance with hot-swappable components, ultimate flexibility for customization, and clear debugging with independent component testing. The foundation that enables Symphony's extension-first philosophy.

**Minimal IDE**

Symphony's philosophy of shipping only essential features in the core (text editor, file tree, syntax highlighting, settings, terminal, extension system) while everything else comes from community extensions. Results in fast startup, zero bloat, and infinite customization.

**Monolithic Kernel Architecture**

One of the five kernel architecture patterns available for system design. All critical components run in kernel space with direct function calls for maximum performance. Offers straightforward design and simplified debugging but creates a single failure domain with rigid structure. Best for high-performance scenarios where maximum speed is critical and stability risks are acceptable.

**Motif** üß©

Another name for Addon‚Äîvisual and experiential extensions on The Grand Stage (External Assembling). Named to fit Symphony's musical metaphor. Includes UI components, themes, custom editors, and enhanced user experiences. **Low policy focus** - primarily concerned with visual presentation and user interaction rather than security or external dependencies.

## N

**Nanokernel Architecture**

One of the five kernel architecture patterns. Uses capability-based security with minimal orchestration. Kernel only grants specific permissions while components self-organize. Offers exceptional security and fine-grained control but requires complex capability management. Perfect for security-critical environments.

Lightweight, deterministic, and free to use (no credits consumed). The rhythm section that keeps workflows flowing smoothly. **Medium policy focus** - handles data transformation and processing with moderate security requirements.

**Orchestra Kit**

The complete framework that assembles, orchestrates, and safeguards Symphony's extension ecosystem. Includes sandboxed execution, dependency management, auto-generated configuration UIs, the marketplace, and the communication protocols between all components. Not just a marketplace‚Äîit's the entire performance system.

**Orchestration**

The intelligent coordination of AI agents, models, and tools to accomplish development tasks. Symphony's core capability‚Äîthe Conductor orchestrates specialized agents to work together like musicians in a symphony.

**Out-of-Process Execution**

The execution model used by The Grand Stage extensions (External Assembling) where each extension runs in an isolated process. Provides crash protection, memory isolation, and independent update cycles. Communication happens through IPC Bus with 0.1-0.5ms latency‚Äîacceptable for user-facing features.

## P

**PDF (Predictability Test Framework)**

A formal testing framework ensuring Symphony's agent-driven pipeline operates reliably and deterministically across multiple runs. Repeatedly executes defined tasks and evaluates whether outputs remain consistent, structured, and within acceptable variance thresholds. Note: The Conductor may use adaptive approaches, simulate alternate paths and perform meta-evaluation.

**Pit, The**

Symphony's Internal Ensemble (IaE) / Internal Assembling‚Äîfive privileged Rust-powered infrastructure extensions that form the unshakable foundation: Pool Manager, DAG Tracker, Artifact Store, Arbitration Engine, and Stale Manager. Runs in-process with the Conductor for maximum performance. The engine room where the orchestra's rhythm section keeps everything synchronized.

**Planner Model** üè†

One of Symphony's seven orchestration models. Designs overall architecture and determines dependencies between components. Creates the structural blueprint that guides implementation.

**Player, The**

A registered extension (Instrument, Operator, or Motif) that has completed formal registration with the Conductor Core and actively participates in orchestrated workflows. Unlike passive extensions that are called manually, Players are authorized performers that commit to governance policies in exchange for intelligent orchestration, optimization, and elevated platform status.

**Polyphony**

Symphony's Melody Marketplace‚Äîa curated registry and community hub for discovering, sharing, and managing Melodies. Features public (community-contributed, rated, open-source workflows), private (organization-specific with controlled access), and featured templates (e.g., ‚ÄúFull Stack Review‚Äù for code pipelines, ‚ÄúAPI Doc Generator‚Äù for docs).

**Pool Manager** üìä

One of The Pit's five infrastructure extensions (Internal Assembling). Symphony's resource maestro that handles AI model lifecycle management‚Äîloading, warming, activation, cooldown. Learns usage patterns to predictively pre-warm models, monitors health, manages costs, and handles failures with automatic fallbacks.

## S

**Sandbox**

The isolated execution environment where user extensions (External Assembling) run. Provides security through permission controls‚Äîextensions can only access resources they explicitly declare and are granted. Prevents malicious code from affecting the core system.

**Solo Mode** üéß

One of Symphony's three operational modes. Ideal for quick help, questions, or code snippets. Uses only lightweight LLM/Enhancer for fast responses without full orchestration overhead.

**SPFR (Symphony Playbook Failure Recovery)**

The complete failure handling strategy for Symphony extensions across all operational scenarios. Defines systematic responses to extension failures based on classification, position in workflow, and system state. Ensures predictable and reliable behavior under all failure conditions.

**Stale Manager** üßπ

One of The Pit's five infrastructure extensions (Internal Assembling). Symphony's digital janitor that continuously optimizes system health. Removes unused data, optimizes storage. Ensures Symphony runs like new even after heavy use.

**Symphony**

The first true AI-First Development Environment (AIDE). A revolutionary platform where intelligent agents orchestrate software creation while humans provide vision and creative direction. Built on a minimal core with infinite extensibility through the Orchestra Kit and dual layer Assembling architecture (Internal + External).

## V

**Virtual DOM**

The technique used by out-of-process extensions to describe their user interfaces. Rust extensions generate Virtual DOM structures that React efficiently renders. Enables extensions to create rich UIs without direct DOM access.

**Virtuoso Mode** üéª

One of Symphony's three operational modes. Ideal for editing, refactoring, or enhancing existing code. Uses editor-centric, context-sensitive models focused on code manipulation rather than creation from scratch.
